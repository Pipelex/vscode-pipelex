# Validation backend — PR #50 code-review round 1: deferred items

Bot review (greptile, chatgpt-codex) on PR #50 (`feature/Use-pipelex-api → dev`). **Fixed:** the cross-file diagnostics path-boundary misroute (`crossFileDiagnostics.ts` — `endsWith(src)` → `endsWith(path.sep + src)`). The items below are **deferred** with reasoning, per the "fix only clear wins; defer doubt" directive.

## Deferred — guard *impossible* scenarios (our own server honors the contract)

- **G5 — "Malformed verdict becomes valid"** (`apiValidationBackend.ts` ~99). The success path casts to the valid arm when `is_valid !== false`. greptile wants `is_valid === true` required explicitly, rejecting unknown shapes. Our backend (`pipelex-api`) always emits `is_valid` as a Pydantic `Literal[True]`/`Literal[False]`, so a body without `is_valid` is a contract violation our reference server cannot produce. Guarding it adds dead branches for an impossible response. Reconsider only if/when a non-Pipelex runtime is targeted.
- **G6 — "Empty invalid report disappears"** (`apiValidationBackend.ts` ~103). The invalid arm forwards `validation_errors` without a non-empty check. The pipelex **structured-info-invariant GATE** guarantees every invalid verdict carries a non-empty `validation_errors[]` (including raw malformed `.mthds`). An empty invalid list is therefore impossible from a conformant server. Same reasoning as G5.

## Deferred — real-but-nuanced (needs a semantics decision, not a quick patch)

- **G4 / codex — "Version probe can hang"** (`apiValidationBackend.ts:75`). `versionGate.ensureCapable()` is awaited before the timed `runWithAbort(validate)`, and `client.version()` carries neither the abort signal nor the request timeout. A server that accepts the connection but stalls on `/version` hangs validation/graph indefinitely. **Real latent bug** (flagged P1 by two bots). The correct fix must preserve the gate's documented "best-effort, never blocks" contract: on timeout it should **skip the gate and proceed to validate**, not abort the analyze (so racing it against `runWithAbort`, which rejects with `AnalyzeAbortError`, is wrong as-is). Wants a small, deliberate change (race the probe against a timeout that *resolves*/skips, with the dangling `version()` left best-effort). Picked up next round.
- **G7 — "Unparsed version bypasses floor"** (`cliValidationBackend.ts` ~159). The CLI floor check only blocks when `getAgentCliVersion()` returns a parsed version; a dev/localized/unparseable `--version` string returns `null` and runs anyway, silently degrading source attribution. This is a fail-open-vs-fail-closed judgment on version detection — fail-closed would break legitimate dev CLIs whose version string is non-standard but new enough. Owner's call; not an obvious win.
- **G8 — "Broken graph output is hidden"** (`cliValidationBackend.ts` ~90). When `--view` succeeds but stdout has no parseable graph JSON, `parseGraphspec()` returns `null` and the caller still reports `ok: true` ("No Graph Available"). Pre-existing graph-path behavior; turning missing graph output into an infra error is a separate UX decision, not part of the 200-diagnostic consume change.
