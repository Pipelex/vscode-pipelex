# Phase 4 code review — fixed + deferred findings

Recall-mode review of the uncommitted Phase 4 backend seam (branch `feature/Use-pipelex-api`), run 2026-06-16. The must-fix correctness items are **fixed** (see below); the rest are **deferred** here with justification. This doc is the follow-up backlog for the seam — pick from it before/after the Phase 5 manual QA.

A later independent review (also 2026-06-16) added one more fix (**#16**) and three deferrals (**#17–#19**), recorded in their respective sections below.

## Fixed in this pass

- **[#1 HIGH] Open graph panel went stale on every non-success save (validation enabled).** The validator only handed the panel an analysis on the `withGraph` success path; the LSP-skip early return, the backend-error catch, and abort all returned without touching the sink, and the panel suppresses its own self-refresh whenever validation is enabled. **Fix:** `GraphAnalysisSink` gained `applyBackendError(uri, err)` and `applySkipped(uri, message)` (both no-op unless the panel is showing `uri`); `MethodGraphPanel` implements them (reusing `renderBackendError` / a notice); `PipelexValidator.onSave` calls `applySkipped` on the LSP-skip path and `applyBackendError` on the catch path. Abort still intentionally does nothing (a newer save is coming). The decline-double-prompt trap is avoided by rendering the caught error directly rather than re-running `analyze()`.
- **[#2 MED-HIGH] The 0.34.0 CLI floor was not enforced on the path that needs it.** `throwIfTooOld` only ran on an unparseable / non-exit-1 spawn failure, so a CLI in `[0.31.0, 0.34.0)` validated cleanly and returned a structured error list *without* `source` / `field_name`, silently degrading cross-file diagnostics with no "too old" warning. **Fix:** `CliValidationBackend.analyze` now calls `throwIfTooOld` up front, before building args / spawning. The version probe is cached per command, so only the first analysis of a session pays for it. The lazy checks in `handleSpawnError` were removed as dead (after the upfront check, the version is either ≥ floor or undeterminable, so they could never throw). New test: a `0.32.0` CLI that would exit 0 is rejected up front and never spawns `validate`. This also restores/extends version gating on the graph path.
- **[#5 MED] LSP-skip early return ran before `cancelInflightByKey`.** A prior in-flight analysis for the same URI was never cancelled and could resolve after the skip cleared diagnostics, re-publishing pipelex diagnostics onto a file with active LSP syntax errors. **Fix:** the per-URI cancel now runs immediately after the `validation.enabled` check, before any early-return guard.
- **[#16 MED-HIGH] Client construction ran outside `analyze`'s try/catch → a misconfigured base URL failed silently.** `getToken()` and `new MthdsApiClient({ baseUrl })` were awaited *before* the `try`, so a non-host-only `pipelex.api.baseUrl` (e.g. a pasted `https://api.pipelex.com/v1` — the client constructor rejects any path/query/credentials) threw a `PipelineRequestError` that escaped `handleError`: `PipelexValidator` saw a non-`BackendError`, cleared diagnostics, and logged to the output channel with **no toast**; the graph panel showed the generic "An error occurred — check the output". Worse, the throw landed *after* the privacy modal was accepted. **Fix:** build the client (token read + constructor) inside a `try`, *before* the privacy gate (both are local, no network), and map a failure via a new `setupError()` — `api-error` with an actionable host-only message for the rejected URL, `infra` for a SecretStorage read failure. The bad-URL case now fails fast (no modal, no request) with a clear "must be host-only … fix `pipelex.api.baseUrl`, or switch `pipelex.backend` to `cli`" message on both consumers. New tests: a `/v1` base URL → `api-error` before prompting/sending; a token-read failure → `infra`.

All vitest tests pass (new regression tests added for #16 and the upfront version gate); `tsc` clean on the changed files (only the pre-existing webview DOM-lib errors remain); the rollup node bundle builds.

## Deferred — correctness (real but lower-severity / narrower trigger)

- **[#3 MED] `isLocalhost` treats `0.0.0.0` as loopback → privacy consent skipped.** `apiValidationBackend.ts` `isLocalhost`. Verified `new URL('http://0.0.0.0:8081').hostname === '0.0.0.0'`. A pasted bind address skips the modal. *Why deferred:* `0.0.0.0` as a client *destination* is unusual; low real-world incidence. *Fix when touched:* drop `0.0.0.0` from the allowlist (loopback is `127.0.0.0/8` + `::1` only). Cheap; bundle with #11.
- **[#11 LOW-MED] IPv6 loopback `[::1]` treated as remote (over-prompts).** Same function. Verified `new URL('http://[::1]:8081').hostname === '[::1]'` (brackets retained), so `host === '::1'` fails. *Fix:* strip surrounding brackets before comparing. Bundle with #3 — both are the loopback-detection helper.
- **[#4 MED] `resolveOwner` `endsWith(src)` can bind a diagnostic to the wrong sibling.** `crossFileDiagnostics.ts`. Verified `'/dir/ba.mthds'.endsWith('a.mthds') === true`; no path-separator boundary. *Why deferred:* the exact `name` / basename arms (the API path always sends `name === source`) usually win first by sort order; the suffix fallback is for CLI relative paths. *Fix:* tighten to `endsWith(path.sep + src)` or drop the suffix arm. Add a regression test with lookalike-suffixed siblings.
- **[#6 MED] A slow-but-healthy API server (> timeout) is reported as "unreachable — is pipelex-api running?".** `apiValidationBackend.runWithAbort`. The timeout reuses the unreachable message; the underlying fetch also keeps running (`client.validate()` takes no `AbortSignal`). *Fix:* distinct "timed out after Nms" message; ideally forward cancellation into the client (needs a small `mthds-js` change to accept a signal on `validate()`).
- **[#7 MED] `/version` capability probe is awaited without an abort/timeout race.** `apiValidationBackend.analyze` → `versionGate.ensureCapable` before the `runWithAbort` wrapper. A hung `/version` stalls the analyze (bounded only by the client's internal poll timeout, not `validation.timeout`) and ignores supersede-aborts. *Fix:* wrap the probe in the same abort/timeout race, or fire it without awaiting (warn-once is already async/best-effort).
- **[#9 MED] Runtime toggle `validation.enabled` false→true (no reload) leaves the graph panel permanently stale.** The `PipelexValidator` is constructed once at activation (only if enabled then); the panel's self-refresh guard reads config live. Toggling on after activation = no validator exists AND the panel skips self-refresh. *Why deferred:* requires a runtime settings toggle without reload; most config changes here are set-and-reload. *Fix:* construct the validator unconditionally and gate its work on the live config, or have the panel track whether a validator is actually wired (e.g. the extension sets a flag) rather than inferring from `validation.enabled`.
- **[#12 LOW-MED] Concurrent saves in the same directory race on the dir-keyed diagnostics store.** `pipelexValidator` — abort is per-URI, `ownedByDir` + the `DiagnosticCollection` are per-directory; the later-finishing run can drop the earlier's sibling diagnostics. *Why deferred:* both runs validate the same directory bundle, so they normally compute the same set (benign); only a mid-edit divergence between the two disk reads bites. *Fix:* key in-flight/abort per directory, or carry a per-run generation token and reject stale applies.
- **[#17 LOW-MED] `confirmRemote` has a TOCTOU on the per-host consent flag → stacked modals on first concurrent use.** `backendFactory.confirmRemote` reads the `globalState` consent key, then `await`s the modal *before* writing it. Two near-simultaneous first-time analyses against the same host both read it unset and each pop their own "Send to API?" modal. *Trigger:* the api backend with no prior consent + a **Save All** of several dirty `.mthds` files (or a save racing an active-editor-switch graph refresh) → N stacked identical modals instead of the documented one-time prompt. *Why deferred:* one-shot, first-use-only, and accepting any one modal sets the flag for the rest of the session; cosmetic annoyance, not a wrong send. *Fix:* dedupe in-flight consent per host with a shared pending `Promise<boolean>` so the first prompt gates concurrent callers.

## Deferred — silent-failure / robustness

- **[#8 MED] Two divergent `BackendError`→UX switches; CLI infra errors are silent in the validator.** `methodGraphPanel.renderBackendError` vs `pipelexValidator.handleBackendError`. For `kind:'infra'` the panel renders the message but the validator only notifies `if (err.userMessage)` — and CLI infra/interpreter errors carry none, so the validator clears diagnostics and stays silent. *Fix:* centralize the kind→message/severity mapping on/near `BackendError` so both consumers render consistently, and give `infra` a default user-facing message.
- **[#13 LOW-MED] `extractJson` only strips `WARNING:` lines.** `cliOutput.ts` — `indexOf('{') … lastIndexOf('}')` spans the whole output; any other stdout/stderr noise containing braces corrupts the slice (graph silently dropped, or a real failure misreported as infra). Pre-existing logic, now load-bearing for both channels. *Fix:* parse the JSON region more defensively (e.g. scan for a balanced object from the first `{`), or have the CLI guarantee JSON-only on the relevant stream.
- **[#18 LOW] `notifyOnce` dedup swallows the auth remedy toast after the first dismissal.** `pipelexValidator.notifyOnce` dedupes on exact message text and only resets `lastNotifiedMessage` on a *successful* validation. For a standing 401/403 (api backend, no key), the "Set API Key" / "Get an API Key" toast appears once; if the user dismisses it without acting, every later save with the same auth error returns early — no toast, no remedy buttons — and no success can reset the dedup (there's no key). *Why deferred:* low severity — the remedy stays reachable via the open graph panel (`renderBackendError` re-renders the buttons each save) and the `Pipelex: Set Hosted API Key` command. *Fix:* don't suppress action-bearing notifications, or reset the dedup once the actionable remedy is surfaced-but-not-taken.

## Resolved later

- **[#10 MED] ✅ RESOLVED (2026-06-17, the 200-diagnostic consumer alignment).** Both fabrication sites are **deleted**, not merged into a shared helper: the runtime's structured-info invariant is now total (pipelex's shared builder emits a non-empty `validation_errors[]` on *every* invalid verdict — a parse-level failure yields one source-less `blueprint_validation` residual; a dry-run failure a graph-level `dry_run` item), so there is nothing left to synthesize. `apiValidationBackend.asValidationOutcome` was removed entirely (the API backend now reads the 200 `is_valid:false` body's `validation_errors[]` directly off `MthdsApiClient.validate()`'s discriminated union — an invalid bundle is a produced verdict, no longer a 422 throw); `cliValidationBackend.parseFailure` dropped its `error_domain==='input'` synthesis (an exit-1 with an empty list is now surfaced as an `infra` `BackendError`, never a fabricated diagnostic). A `dry_run` item flows through verbatim and lands on the primary file (no `source`) via the existing `crossFileDiagnostics` fallback. Tests updated: the two "synthesizes a single diagnostic" cases became "consumes a `dry_run` item directly" + "empty list → infra". The honest-category question is moot — categories now come from the runtime, never the consumer.

## Deferred — cleanup / efficiency (no behavior bug)
- **[#14 LOW] `gatherBundleFiles` runs on every save, including the valid path.** `pipelexValidator.onSave`. The contents are only consumed by `buildBundleDiagnostics` on failure; the default CLI backend then re-reads the siblings via `--library-dir`. *Fix:* defer the gather into the failure branch (or gate on backend kind, as the graph panel already does).
- **[#15 LOW] Dead method `BackendFactory.backendKind()`** — public, never called, duplicates `getBackend`'s decision. *Fix:* remove it (or wire the intended "messaging" caller).
- **[#19 LOW] Three duplicated URL-host helpers.** `isLocalhost` (`apiValidationBackend.ts`), `isHostedPipelexApi` (`apiVersionGate.ts`), and `hostOf` (`backendFactory.ts`) each re-implement parse-URL-and-extract-host with their own try/catch and host rules — and that duplication is the structural reason the loopback bugs (#3/#11) live in only one of them. *Fix:* extract one shared host util (parse + bracket-strip + classify loopback/hosted) and route all three through it; closes #3/#11 at the root instead of per-helper.

## Not bugs (verified, recorded so they aren't re-litigated)

- Brand boundary (MTHDS vs Pipelex) — **clean**: `PipelexValidationReport` / `graph_spec` / `mthds_names` and the `pipelex.*` settings are correctly neutral or runtime-envelope names.
- `docs/features/validation-backends.md` — no hard-wraps, no hardcoded counts.
- `extractCodeFromMessage`'s `\w+` capture and `field_path.split('.').pop()!` are pre-existing edges unlikely to fire with snake_case identifiers; abort correctly does not refresh the panel (a newer save supersedes it).
- The 200-`validated_pipes`-FAILURE channel and buffer-aware sibling gathering remain deliberate out-of-scope follow-ups (see the plan doc).
- `applyAnalysis`'s `'No Graph Available'` branch for an `{ok:false, errors:[]}` outcome — **unreachable**: both `asValidationOutcome` (API) and `parseFailure` (CLI) only ever return `ok:false` with a non-empty `errors`, else they throw a `BackendError`. A valid bundle with no `main_pipe` is `ok:true, graph:null`, which is the legitimate "No Graph Available" case.
- The API graph panel still calling the backend on save when `validation.enabled=false` — **by design**: the panel renders the graph the user explicitly opened, so it needs the backend regardless of the validation toggle; the privacy modal still gates the first remote send.
- `setHtml` resetting `webviewReady=false` and the `pendingData` overwrite on back-to-back same-URI renders — **correct**: the later payload supersedes the earlier for the same file (latest-wins is the intended behavior).
