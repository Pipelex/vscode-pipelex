## Fixed in this pass

- **[#1 HIGH] Open graph panel went stale on every non-success save (validation enabled).** The validator only handed the panel an analysis on the `withGraph` success path; the LSP-skip early return, the backend-error catch, and abort all returned without touching the sink, and the panel suppresses its own self-refresh whenever validation is enabled. **Fix:** `GraphAnalysisSink` gained `applyBackendError(uri, err)` and `applySkipped(uri, message)` (both no-op unless the panel is showing `uri`); `MethodGraphPanel` implements them (reusing `renderBackendError` / a notice); `PipelexValidator.onSave` calls `applySkipped` on the LSP-skip path and `applyBackendError` on the catch path. Abort still intentionally does nothing (a newer save is coming). The decline-double-prompt trap is avoided by rendering the caught error directly rather than re-running `analyze()`.
- **[#2 MED-HIGH] The 0.34.0 CLI floor was not enforced on the path that needs it.** `throwIfTooOld` only ran on an unparseable / non-exit-1 spawn failure, so a CLI in `[0.31.0, 0.34.0)` validated cleanly and returned a structured error list *without* `source` / `field_name`, silently degrading cross-file diagnostics with no "too old" warning. **Fix:** `CliValidationBackend.analyze` now calls `throwIfTooOld` up front, before building args / spawning. The version probe is cached per command, so only the first analysis of a session pays for it. The lazy checks in `handleSpawnError` were removed as dead (after the upfront check, the version is either â‰Ą floor or undeterminable, so they could never throw). New test: a `0.32.0` CLI that would exit 0 is rejected up front and never spawns `validate`. This also restores/extends version gating on the graph path.
- **[#5 MED] LSP-skip early return ran before `cancelInflightByKey`.** A prior in-flight analysis for the same URI was never cancelled and could resolve after the skip cleared diagnostics, re-publishing pipelex diagnostics onto a file with active LSP syntax errors. **Fix:** the per-URI cancel now runs immediately after the `validation.enabled` check, before any early-return guard.

All 142 vitest tests pass (+1 new); `tsc` clean on the changed files (only the pre-existing webview DOM-lib errors remain); the rollup node bundle builds.

- **[#6 MED] A slow-but-healthy API server (> timeout) is reported as "unreachable â€” is pipelex-api running?".** `apiValidationBackend.runWithAbort`. - **[#9 MED] Runtime toggle `validation.enabled` falseâ†’true (no reload) leaves the graph panel permanently stale.** The `PipelexValidator` is constructed once at activation (only if enabled then); the panel's self-refresh guard reads config live. Toggling on after activation = no validator exists AND the panel skips self-refresh. *Why deferred:* requires a runtime settings toggle without reload; most config changes here are set-and-reload. *Fix:* construct the validator unconditionally and gate its work on the live config, or have the panel track whether a validator is actually wired (e.g. the extension sets a flag) rather than inferring from `validation.enabled`.
